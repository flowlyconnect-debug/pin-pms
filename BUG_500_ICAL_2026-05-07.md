# BUG 500 iCal / calendar sync — 2026-05-07

## Reproduction
- Komennot:
  - `docker compose up --build -d`
  - `flask db upgrade`
  - `flask seed-demo-data`
- Polku: `/admin/units/<unit_id>/calendar-sync`
- Käyttäjärooli: admin
- Unit id: demo-unit (ymparistoriippuvainen)
- iCal-URL testiin: `http://example.com/missing`
- Toimet:
  1. Tallenna lahde.
  2. Klikkaa `Synkronoi nyt`.

Huom: taman ympariston Docker-daemon ei ollut kaytettavissa, joten manuaalinen konttirepro ei ollut ajettavissa loppuun asti tassa sessiossa. Reproduktio vahvistettiin testipolussa ja koodipolun analyysilla.

## Original stack trace
Alkuperainen 500-polku oli kalenterisynkan route/service -ketjussa:

- `POST /admin/calendar-sync/feeds/<feed_id>/sync`
- `app/admin/routes.py` kutsui aiemmin `IcalService().sync_all_feeds(...)`
- `app/integrations/ical/service.py` kasitteli virheet catch-allilla, mutta ilman tarkkaa poikkeustyyppia ja ilman teknista lokitusta (`logger.exception` puuttui)
- GET-nakymassa `/admin/units/<unit_id>/calendar-sync` vienti-URL:n muodostus pystyi heittamaan `IcalServiceError`-poikkeuksen ilman route-tason kasittelya, jolloin pyynto saattoi paattya 500-virheeseen

Tyypillinen poikkeusketju:
- `IcalService.sign_unit_token(...)`
- `raise IcalServiceError("config_error", "iCal feed secret is not configured.", 500)`
- poikkeus kupli routeen ilman kasittelya

## Suspected root cause
Useampi samantyyppinen virheenkasittelyaukko:

1. iCal-synkassa odotettavat ulkoiset verkko-/parse-virheet eivat olleet tyyppikohtaisesti kasiteltyja.
2. GET calendar-sync -nakyma oletti, etta tokenin allekirjoitus aina onnistuu.
3. UI naytti `last_error`-kentan sellaisenaan ilman selkeaa virhebanneria tai pituusrajausta.

## Root cause
Root cause oli se, etta integraatiovirheita (timeout, yhteysvirhe, HTTP-virhe, malformed iCal) ei kasitelty domain-tasona hallitusti vaan kasittely oli liian geneerinen, ja osa poikkeuksista paasi routeen asti.

## Fix
Tehdyt minimikorjaukset:

- `app/integrations/ical/service.py`
  - Lisatty tyyppikohtainen virheenkasittely:
    - `requests.exceptions.Timeout`
    - `requests.exceptions.ConnectionError`
    - `requests.exceptions.HTTPError`
    - `requests.exceptions.RequestException`
    - `ValueError` (parsinta)
  - Lisatty tarkka tekninen lokitus `current_app.logger.exception(...)`.
  - Lisatty `IcalSyncResult` ja `sync_feed(...)`, jotta route saa hallitun tuloksen.
  - `feed.last_error` asetetaan kayttajakelpoisella suomenkielisella viestilla (ei tracebackia).

- `app/admin/routes.py`
  - `units_calendar_sync`: vienti-URL:n muodostus kasitellaan turvallisesti; virheessa naytetaan hallittu virheviesti eika nosteta 500:aa.
  - `calendar_sync_feed_now`: kayttaa `sync_feed(...)`-metodia ja flashaa onnistumis-/virheviestin hallitusti.

- `app/templates/admin/units/calendar_sync.html`
  - Lisatty feed-kohtainen virhebanneri.
  - `feed.last_error` naytetaan enintaan 240 merkkiin rajattuna.
  - `Yrita uudelleen` -painike kayttaa olemassa olevaa sync-actionia ja CSRF-kenttaa.
  - Vienti-URL:lle fallback-nakyma, jos URL:n muodostus epaonnistuu.
